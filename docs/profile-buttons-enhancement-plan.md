# План доработки экрана профиля: добавление кнопок навигации

## Обзор

Добавить на экран профиля (`ProfileRootScreen`) кнопки навигации аналогично iOS-приложению. При нажатии на кнопки пока что должно происходить логирование в консоль (без реальной навигации). Все данные для кнопок уже доступны в модели `User`.

## Назначение

Сделать Android-версию экрана профиля консистентной с iOS-версией, добавив недостающие кнопки для навигации к функционалу приложения.

## Что сделано

- ✅ `ProfileRootScreen` - базовая реализация с отображением профиля и кнопкой выхода
- ✅ `ProfileViewModel` - загружает профиль пользователя, страну и город
- ✅ `User` модель - содержит все необходимые данные для кнопок:
  - `friendRequestCount` - количество заявок в друзья
  - `friendsCount` - количество друзей
  - `parksCount` (area_count) - количество площадок, где тренируется
  - `addedParks` - список добавленных площадок
  - `journalCount` - количество дневников
  - Вычисляемые свойства: `hasJournals`, `hasFriends`, `hasUsedParks`, `hasAddedParks`

## Рефакторинг

### Архитектурные замечания

Текущий план вызывает `appContainer?.logoutUseCase` прямо из UI (`ProfileRootScreen`). Это соответствует текущему коду в файле, поэтому для текущей задачи это допустимо.

Однако, в идеальной архитектуре (MVVM), метод `logout()` должен находиться в `ProfileViewModel`, и UI должен вызывать `viewModel.logout()`. Это сделало бы код чище и проще для тестирования, но это выходит за рамки текущей задачи по добавлению алерта.

**Рекомендуемый подход (для будущих улучшений):**

```kotlin
// В ProfileViewModel
fun logout() {
    viewModelScope.launch {
        logoutUseCase?.invoke()
    }
}

// В ProfileRootScreen
AlertDialog(
    confirmButton = {
        TextButton(
            onClick = {
                viewModel.logout()
                showLogoutDialog = false
            }
        ) {
            Text(text = stringResource(id = R.string.logout))
        }
    }
)
```

**Преимущества этого подхода:**
- UI не зависит напрямую от `appContainer` и `logoutUseCase`
- Логика логаута инкапсулирована в ViewModel
- Проще тестировать (можно мокать ViewModel вместо всего контейнера)
- Соответствует MVVM принципам (UI → ViewModel → Use Case)

**Примечание:** Этот рефакторинг может быть выполнен в отдельной задаче после завершения текущего этапа.
