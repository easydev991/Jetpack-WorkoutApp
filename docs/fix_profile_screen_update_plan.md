# План исправления проблемы обновления экрана профиля после авторизации

## Описание проблемы

После успешной авторизации через `LoginSheetHost`:
- Пользователь успешно сохраняется (видно в логах: `Пользователь сохранён: 280084`)
- При перезапуске приложения пользователь загружается (видно в логах: `Текущий пользователь изменился: 280084`)
- **НО** на экране профиля отображается кнопка "Авторизоваться", а не профиль пользователя

## Корневая причина

### 1. Отсутствие механизма автоматической загрузки профиля при изменении пользователя

ProfileViewModel не подписывался на изменения `currentUser` Flow, поэтому при перезапуске приложения и обновлении пользователя данные страны/города не загружались автоматически в `ProfileUiState`.

### 2. Необходимость загрузки профиля с сервера после авторизации

При успешной авторизации нужно загрузить свежие данные пользователя с сервера. Данные профиля должны загружаться из кэша через `currentUser` Flow, но нет гарантии, что Flow успел обновиться до навигации на вкладку профиля.

### 3. Метод `loadProfile()` должен загружать только страну и город

`ProfileViewModel` должен загружать только страну и город из локального репозитория стран. Полный профиль пользователя загружается через `swRepository.getUser(userId)` и сохраняется в кэш.

## Решение

**Идея:** ProfileViewModel автоматически подписывается на изменения `currentUser` Flow и вызывает `loadProfileAddress(user)` при каждом изменении. После успешной авторизации `RootScreen` явно вызывает `loadProfileFromServer(userId)` для загрузки свежих данных с сервера.

**Цепочка изменений:** `LoginViewModel` → `LoginScreen` → `LoginSheetHost` → `RootScreen` → `ProfileViewModel`

**Ключевые изменения:**
1. ProfileViewModel содержит init-блок с подпиской на currentUser Flow
2. Метод `loadProfile(user)` переименован в `loadProfileAddress(user)` и стал private
3. Метод `loadProfileFromServer(userId)` загружает пользователя с сервера, затем вызывает `loadProfileAddress(user)`
4. ProfileViewModel создается на уровне RootScreen для единого экземпляра

## Примечание

**Почему это решает проблему:**

1. **Автоматическая подписка на изменения пользователя:** ProfileViewModel содержит init-блок с подпиской на currentUser Flow, который вызывает loadProfileAddress(user) при каждом изменении.
2. **Явная загрузка профиля с сервера:** После авторизации RootScreen вызывает loadProfileFromServer(userId), который загружает пользователя с сервера через swRepository.getUser(userId).
3. **Цепочка передачи userId:** LoginViewModel → LoginScreen → LoginSheetHost → RootScreen → ProfileViewModel.
4. **Метод loadProfileFromServer(userId):** Загружает профиль с сервера, затем вызывает loadProfileAddress(user) для загрузки страны и города.
5. **Метод loadProfileAddress(user):** Приватный метод, загружает только страну и город из CountriesRepository.

**Преимущества подхода:**
- Автоматическая загрузка профиля при перезапуске приложения через init-подписку
- Явная загрузка профиля с сервера после авторизации, без зависимости от асинхронных обновлений Flow
- Гарантированное обновление UI после авторизации
- Простая и понятная цепочка передачи данных
- Соответствие архитектуре MVVM (загрузка данных в ViewModel)
- Нет Mutex и флагов - простая архитектура
