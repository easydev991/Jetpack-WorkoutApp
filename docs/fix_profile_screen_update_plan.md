# План исправления проблемы обновления экрана профиля после авторизации

## Описание проблемы

После успешной авторизации через `LoginSheetHost`:
- Пользователь успешно сохраняется (видно в логах: `Пользователь сохранён: 280084`)
- При перезапуске приложения пользователь загружается (видно в логах: `Текущий пользователь изменился: 280084`)
- **НО** на экране профиля отображается кнопка "Авторизоваться", а не профиль пользователя

## Корневая причина

### 1. Пустой метод `reloadProfile()` в ProfileViewModel

Метод `reloadProfile()` был пустой и НЕ ДЕЛАЛ НИЧЕГО. Даже если `RootScreen` вызывал его после авторизации, данные профиля не загружались с сервера.

### 2. Отсутствие механизма загрузки профиля после авторизации

При успешной авторизации вызывался пустой метод `reloadProfile()`. Данные профиля должны загружаться из кэша через `currentUser` Flow, но нет гарантии, что Flow успел обновиться до навигации на вкладку профиля.

### 3. Отсутствие явной загрузки профиля с сервера

`ProfileViewModel` подписывается на `currentUser` в `init`, но это **асинхронный процесс**. И даже если навигация происходит быстро, метод `loadProfile()` загружает только **страну и город** из кэша, а не весь профиль пользователя с сервера.

## Решение

### Реализованный подход: Явная загрузка профиля с сервера после авторизации

**Идея:** После успешной авторизации передавать `userId` из `LoginSheetHost` в `ProfileViewModel`, чтобы она явно загрузила профиль пользователя с сервера.

**Цепочка изменений:** `LoginViewModel` → `LoginScreen` → `LoginSheetHost` → `RootScreen` → `ProfileViewModel`

### Этап 8: Устранение дубликатной загрузки профиля ✅

**Проблема:** При авторизации профиль загружается 3 раза.

**Анализ причин:**
1. **Первая загрузка:** `init` подписывается на `currentUser` Flow → `loadProfile(user)` → "Профиль загружен"
2. **Вторая загрузка:** `RootScreen` вызывает `loadUserProfileFromServer(userId)` → "Профиль загружен с сервера"
3. **Третья загрузка:** После сохранения в кэш `currentUser` Flow обновляется → `loadProfile(user)` снова

**Решение:**
- [x] Добавить флаг `isExplicitlyLoadingProfile` для предотвращения повторной загрузки
- [x] В `init` проверять флаг перед вызовом `loadProfile`
- [x] В `loadUserProfileFromServer` устанавливать и сбрасывать флаг

**Реализация:**
1. Добавлен флаг `isExplicitlyLoadingProfile` в `ProfileViewModel`
2. В `init` блоке при подписке на `currentUser` Flow добавлена проверка флага
3. В `loadUserProfileFromServer` флаг устанавливается в `true` перед загрузкой и сбрасывается в `false` после завершения

**Результат:**
- ✅ Профиль загружается только 1 раз при авторизации
- ✅ Флаг предотвращает дубликатную загрузку через Flow во время явной загрузки с сервера

## Примечание

**Почему это решает проблему:**

1. **Явная загрузка профиля с сервера:** Вместо ожидания обновления `currentUser` Flow, мы явно загружаем профиль с сервера по `userId`
2. **Цепочка передачи userId:** `LoginViewModel` → `LoginScreen` → `LoginSheetHost` → `RootScreen` → `ProfileViewModel`
3. **Метод `loadUserProfileFromServer(userId)`:** Загружает профиль с сервера, затем вызывает `loadProfile(user)` для загрузки страны и города
4. **Навигация на вкладку профиля:** Гарантирует пересоздание UI и отображение обновленных данных

**Преимущества подхода:**
- Явная загрузка профиля с сервера, без зависимости от асинхронных обновлений Flow
- Гарантированное обновление UI после авторизации
- Простая и понятная цепочка передачи данных
- Соответствие архитектуре MVVM (загрузка данных в ViewModel)
