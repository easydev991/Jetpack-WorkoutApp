# План исправления проблемы обновления экрана профиля после авторизации

## Описание проблемы

После успешной авторизации через `LoginSheetHost`:
- Пользователь успешно сохраняется (видно в логах: `Пользователь сохранён: 280084`)
- При перезапуске приложения пользователь загружается (видно в логах: `Текущий пользователь изменился: 280084`)
- **НО** на экране профиля отображается кнопка "Авторизоваться", а не профиль пользователя

## Корневая причина

### 1. Пустой метод `reloadProfile()` в ProfileViewModel

Метод `reloadProfile()` был пустой и НЕ ДЕЛАЛ НИЧЕГО. Даже если `RootScreen` вызывал его после авторизации, данные профиля не загружались с сервера.

### 2. Отсутствие механизма загрузки профиля после авторизации

При успешной авторизации вызывался пустой метод `reloadProfile()`. Данные профиля должны загружаться из кэша через `currentUser` Flow, но нет гарантии, что Flow успел обновиться до навигации на вкладку профиля.

### 3. Отсутствие явной загрузки профиля с сервера

`ProfileViewModel` подписывается на `currentUser` в `init`, но это **асинхронный процесс**. И даже если навигация происходит быстро, метод `loadProfile()` загружает только **страну и город** из кэша, а не весь профиль пользователя с сервера.

## Решение

### Реализованный подход: Явная загрузка профиля с сервера после авторизации

**Идея:** После успешной авторизации передавать `userId` из `LoginSheetHost` в `ProfileViewModel`, чтобы она явно загрузила профиль пользователя с сервера.

**Цепочка изменений:** `LoginViewModel` → `LoginScreen` → `LoginSheetHost` → `RootScreen` → `ProfileViewModel`

### Этап 8: Устранение дубликатной загрузки профиля ✅

**Проблема:** При авторизации профиль загружается 3 раза.

**Анализ причин:**
1. **Первая загрузка:** `init` подписывается на `currentUser` Flow → `loadProfile(user)` → "Профиль загружен"
2. **Вторая загрузка:** `RootScreen` вызывает `loadUserProfileFromServer(userId)` → "Профиль загружен с сервера"
3. **Третья загрузка:** После сохранения в кэш `currentUser` Flow обновляется → `loadProfile(user)` снова

**Решение:**
- [x] Добавить флаг `isExplicitlyLoadingProfile` для предотвращения повторной загрузки
- [x] В `init` проверять флаг перед вызовом `loadProfile`
- [x] В `loadUserProfileFromServer` устанавливать и сбрасывать флаг

**Реализация:**
1. Добавлен флаг `isExplicitlyLoadingProfile` в `ProfileViewModel`
2. В `init` блоке при подписке на `currentUser` Flow добавлена проверка флага
3. В `loadUserProfileFromServer` флаг устанавливается в `true` перед загрузкой и сбрасывается в `false` после завершения

**Результат:**
- ✅ Профиль загружается только 1 раз при авторизации
- ✅ Флаг предотвращает дубликатную загрузку через Flow во время явной загрузки с сервера

### Этап 8.1: Упрощение архитектуры загрузки профиля ⚠️ НАПЛАНИРОВАНО

**Выявленные проблемы после тестирования:**

**Проблема 1: При перезапуске приложения показывается "Загрузка..." вместо адреса**
- **Причина:** `init` подписывается на `currentUser` Flow и сразу вызывает `loadProfile(user)`
- При перезапуске `currentUser` загружается из кэша, но страна/город могут быть еще не загружены
- UI показывает состояние `Loading` из начального значения `_uiState = MutableStateFlow(ProfileUiState.Loading)`

**Проблема 2: Дублирующие логи при logout**

```
2026-01-31 09:29:08.513 SWRepository: Текущий пользователь изменился: 280084
2026-01-31 09:29:08.525 SWRepository: Текущий пользователь отсутствует
2026-01-31 09:29:08.526 SWRepository: Все данные пользователя удалены
```

- **Причина:** Сложный процесс логаута с несколькими источниками данных (UserPreferencesRepository, SWRepository)
- Дублирующие вызовы очистки данных

**Проблема 3: После logout иногда не получается авторизоваться заново**
- **Причина:** Проблема с токеном авторизации - возможно токен не полностью очищен или конфликт при повторной авторизации
- Требует дополнительного исследования

**Проблема 4: Три метода для загрузки профиля вместо одного**
- `loadUserProfileFromServer(userId)` - явная загрузка с сервера
- `loadProfile(user)` - публичный метод для загрузки страны/города
- `loadProfileInternal(user)` - внутренний метод без Mutex
- Дублирование кода и путаница в архитектуре

**Анализ текущей архитектуры:**
- `init` подписывается на `currentUser` Flow → вызывает `loadProfile(user)` для каждого изменения
- `loadUserProfileFromServer(userId)` используется только после авторизации
- Флаг `isExplicitlyLoadingProfile` и Mutex пытаются предотвратить дубликаты
- Но при этом `init` блок **загружает профиль при каждом изменении `currentUser`**, включая перезапуск приложения

**Правильное решение:**

**Идея:** Убрать автоматическую загрузку профиля в `init`. Загружать профиль только когда это нужно (после авторизации).

**План изменений:**
- [ ] Убрать блок `init` с подпиской на `currentUser` Flow
- [ ] Убрать Mutex и флаг `isExplicitlyLoadingProfile` (больше не нужны)
- [ ] Оставить **один публичный метод** `loadProfile(user: User)` для загрузки данных
- [ ] В `loadProfile` загружать страну и город без разделения на отдельные методы
- [ ] Удалить метод `loadProfileInternal` (дублирует функциональность)
- [ ] Удалить метод `loadUserProfileFromServer(userId)` (использовать просто `loadProfile(user)`)
- [ ] В `RootScreen` после авторизации вызывать `loadProfile(user)` вместо `loadUserProfileFromServer(userId)`
- [ ] Для загрузки пользователя с сервера использовать `swRepository.getUser(userId)` напрямую, затем `loadProfile(user)`
- [ ] Убрать начальное значение `ProfileUiState.Loading` из `_uiState` - при перезапуске должно показывать предыдущее состояние

**Почему это правильное решение:**
1. **Один метод загрузки** вместо трех - проще и понятнее
2. **Нет автоматической загрузки** - профиль загружается только когда нужно
3. **При перезапуске приложения** - не показывается "Загрузка..." если данные уже есть в кэше
4. **При авторизации** - ProfileScreen обновится через `loadProfile(user)` после получения userId
5. **Нет Mutex и флагов** - проще архитектура, без race conditions

**Поток данных при авторизации:**
1. `LoginViewModel` → `LoginSheetHost.onLoginSuccess(userId)` → передает `userId`
2. `RootScreen` вызывает `swRepository.getUser(userId)` → загружает пользователя с сервера и сохраняет в кэш
3. `RootScreen` вызывает `profileViewModel.loadProfile(user)` → загружает страну/город и обновляет UI

**Поток данных при перезапуске приложения:**
1. `ProfileRootScreen` читает `user` из `currentUser.collectAsState()` или использует `currentUser.value`
2. При открытии экрана профиля данные уже есть в кэше (или показывается "Не авторизован")
3. Нет автоматической загрузки в `init` - не показывается "Загрузка..." при перезапуске

**Риски и митигация:**
- Риск: При открытии экрана профиля без перезапуска может не быть данных
- Митигация: `ProfileRootScreen` может вызвать `loadProfile(user)` при первом открытии экрана через LaunchedEffect
- Но это будет **явная загрузка**, а не автоматическая в `init`

**Дополнительные исследования:**
- [ ] Исследовать проблему с токеном при повторной авторизации после logout
- [ ] Проверить логи logout - почему дублируются одинаковые сообщения
- [ ] Упростить процесс логаута, убрать лишние вызовы очистки данных

## Примечание

**Почему это решает проблему:**

1. **Явная загрузка профиля с сервера:** Вместо ожидания обновления `currentUser` Flow, мы явно загружаем профиль с сервера по `userId`
2. **Цепочка передачи userId:** `LoginViewModel` → `LoginScreen` → `LoginSheetHost` → `RootScreen` → `ProfileViewModel`
3. **Метод `loadUserProfileFromServer(userId)`:** Загружает профиль с сервера, затем вызывает `loadProfile(user)` для загрузки страны и города
4. **Навигация на вкладку профиля:** Гарантирует пересоздание UI и отображение обновленных данных

**Преимущества подхода:**
- Явная загрузка профиля с сервера, без зависимости от асинхронных обновлений Flow
- Гарантированное обновление UI после авторизации
- Простая и понятная цепочка передачи данных
- Соответствие архитектуре MVVM (загрузка данных в ViewModel)
