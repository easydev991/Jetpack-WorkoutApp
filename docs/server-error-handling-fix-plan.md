## Проблема

**Статус:** ✅ **РЕШЕНО** - Восстановление пароля работает корректно

При отправке запроса для восстановления пароля с несуществующими данными (например, цифру "1") сервер возвращает ответ:

```json
{
    "errors": [
        "Не найден пользователь с таким логином или e-mail"
    ]
}
```

**Ожидаемое поведение:** Пользователь должен видеть текст ошибки из поля `errors` ("Не найден пользователь с таким логином или e-mail").

**Фактическое поведение:** ✅ РЕШЕНО - Пользователь видит правильный текст ошибки.

**Корневая причина:** Проблема была решена путем добавления детального логирования и проверки работы `ErrorResponse.realMessage`.

## Детальный анализ iOS-реализации

### 1. ErrorResponse.swift (строки 31-37)

```swift
var realMessage: String? {
    if let message {
        message
    } else {
        errors.isEmpty ? nil : errors.joined(separator: ", ")
    }
}
```

**Приоритет:** `message` → `errors` → `null`
**Ключевой момент:** Проверка `if let message` возвращает `false`, если `message` равен `nil`, но возвращает `true`, если `message` это пустая строка `""`.

### 2. APIError.swift (строки 16-26)

```swift

    let serverCode = error.makeRealCode(statusCode: statusCode)

    // Приоритет 1: Кастомные сообщения из ErrorResponse
    if let message = error.realMessage {
        self = .customError(
            code: serverCode,
            message: message.trimmingCharacters(in: .whitespacesAndNewlines)
        )
        return
    }
    // ...
}
```

**Приоритет:** `error.realMessage` → стандартные коды ошибок

## Возможные причины проблемы

### Причина 1: Десериализация пустой строки вместо null

При десериализации JSON ответа с kotlinx.serialization может произойти следующее:
- В iOS: `message` отсутствует в JSON → декодер инициализирует его как `nil` → `if let message` возвращает `false` → используется `errors`
- В Android: `message` отсутствует в JSON → декодер инициализирует его как `""` (пустая строка) → `message ?: errors` возвращает `message` (пустую строку) → используется пустая строка вместо `errors`

**Проверка:** В iOS используется `decodeIfPresent` (строка 12):

```swift
self.message = try container.decodeIfPresent(String.self, forKey: .message)
```

Это гарантирует, что если поле отсутствует в JSON, оно будет `nil`.

В Android используется `val message: String? = null` (строка 18) - по умолчанию `null`, но нужно проверить поведение kotlinx.serialization.

### Причина 2: Проблема с настройками JSON-десериализатора

В `SWRepository.kt` (строки 160-163):

```kotlin
private val json = Json {
    ignoreUnknownKeys = true
    isLenient = true
}
```

Флаг `isLenient = true` может вызывать неожиданное поведение при десериализации.

### Причина 3: Тело ответа не десериализируется корректно

Тело ответа может содержать пробелы или переносы строк, которые мешают десериализации.

## Дополнительные заметки

### Правила разработки проекта

- **Безопасное разворачивание опционалов:** Не использовать `!!`, использовать `?.`, `?:`, `checkNotNull`
- **Логирование на русском:** Все логи писать на русском языке
- **TDD:** Писать тесты перед реализацией, если это возможно
- **Форматирование:** Выполнять `make format` после изменений в коде

### Архитектурные принципы

- **Обработка ошибок:** Использовать `ServerException` и `NetworkException` для типов исключений
- **Data Layer:** `ErrorResponse` - модель для десериализации ошибок от сервера
- **Domain Layer:** `ServerException` и `NetworkException` - исключения для бизнес-логики
- **UI Layer:** ViewModels обрабатывают исключения и обновляют UI state

### Ожидаемый результат

После исправления:
1. При ошибке сервера с полем `errors` пользователь видит текст ошибки из этого поля
2. При ошибке сервера без поля `errors` используется сообщение из `APIError.fromStatusCode`
3. Обработка ошибок работает одинаково для всех API запросов
4. Тесты подтверждают корректность работы
