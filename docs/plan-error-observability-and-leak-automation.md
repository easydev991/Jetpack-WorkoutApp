# План доработок: наблюдаемость ошибок и автоматизация контроля утечек

## Цель

- [ ] Построить единый и полный контур отлова ошибок в приложении (fatal + non-fatal), чтобы ошибки не терялись в `Log.e`.
- [ ] Гарантировать, что ошибки из `ViewModel` и `Data` слоев доходят до `Logger` и Firebase Crashlytics.
- [ ] Внедрить простую, регулярную и максимально автоматизированную проверку утечек памяти (LeakCanary + автосценарии), чтобы ручные проверки были только исключением.

## Контекст и исходные проблемы (по текущему состоянию)

- [x] Для `ChatViewModel` прямые `Log.e` заменены на единый маршрут `Logger + CrashReporter`.
- [ ] В части остальных ViewModel/feature-кода еще может использоваться `Log.*` напрямую, без единого маршрута в `Logger`/`CrashReporter`.
- [ ] В release-режиме `NoOpLogger` снижает шум в логах, но часть non-fatal ошибок может быть недостаточно наблюдаемой, если не отправлена в Crashlytics.
- [ ] Для утечек нет постоянного автоматического контура; есть рисковые зоны вокруг карты и долгоживущих ресурсов.
- [ ] `lintDebug` уже сигналит `StaticFieldLeak` и другие warning/error, но это не встроено в обязательный релизный gate.

## Принципы реализации

- [ ] TDD-first: сначала тесты/сценарии, потом изменения.
- [ ] Единая точка ошибок: любой catch-блок должен иметь предсказуемый маршрут ошибки.
- [ ] Минимум ручной рутины: nightly/PR автопроверки и понятный отчет.
- [ ] Без переусложнения: решение под маленькую аудиторию и бесплатное приложение.

## Архитектурное решение

- [ ] Зафиксировать policy: `Log.*` в feature-коде не используется для обработки ошибок.
- [ ] Ошибка из `ViewModel` проходит через `UserNotifier` и/или `Logger`.
- [ ] Техническая ошибка (exception/throwable) дополнительно отправляется в `CrashReporter` как non-fatal.
- [ ] В data/domain слоях оставить текущие `Result<T>` и усилить единообразие логирования/репортинга.
- [ ] Для утечек использовать `LeakCanary` в debug/internal и автоматический тестовый прогон, который валит пайплайн при подтвержденных leak-детекциях.

## Этап 0: Definition of Done и чек-лист

- [ ] Описать и согласовать DoD для ошибки:
- [ ] Любой `catch` не должен заканчивать путь только `Log.e`.
- [ ] Для ожидаемых ошибок: `Result.failure` + UI уведомление.
- [ ] Для неожиданных ошибок: `logger.e` + `crashReporter.logException`.
- [ ] Описать и согласовать DoD для утечек:
- [ ] PR не проходит при новых leak-регрессиях в обязательных сценариях.
- [ ] Для релиза есть отчет по утечкам (автоматический артефакт).

## Этап 1: Инвентаризация ошибок и точек потерь (Red)

- [ ] Собрать список всех мест с `Log.e`/`Log.w` в `ViewModel` и `UI`, где нет маршрута в `Logger`/`CrashReporter`.
- [ ] Сгруппировать по критичности:
- [ ] P0: авторизация, сообщения, профиль, создание/редактирование контента.
- [ ] P1: вторичные экраны и вспомогательные операции.
- [ ] Подготовить таблицу миграции: файл, тип ошибки, текущий путь, целевой путь.
- [ ] Зафиксировать baseline метрики Crashlytics (до изменений):
- [ ] Количество fatal/non-fatal за 7 дней.
- [ ] Crash-free users/sessions.

## Этап 2: Контракт ошибок (Green)

- [ ] Ввести/уточнить единый guideline по обработке:
- [ ] `IOException` и `HttpException` отдельно.
- [ ] `CancellationException` всегда rethrow.
- [ ] Неожиданное `Exception` репортится в Crashlytics.
- [ ] Утвердить стандарт для ViewModel:
- [ ] `logger.e(...)` через интерфейс `Logger`.
- [ ] `userNotifier.handleError(...)` для UI.
- [ ] `crashReporter.logException(...)` для non-fatal технических сбоев.
- [ ] Добавить документированную матрицу "тип ошибки -> действие":
- [ ] Network/Server/Validation/Generic/Location.

## Этап 3: Миграция feature-кода (ChatViewModel и далее)

- [x] `ChatViewModel` переведен на единый маршрут ошибок: без прямого `Log.e`, с `Logger + CrashReporter` и сохранением UX через `UserNotifier`.
- [ ] По аналогии пройтись по остальным ViewModel с прямыми `Log.*`.
- [x] Проверено: `markAsRead`, `sendMessage`, `refresh` больше не "тихие", дубли в UI/Crashlytics отсутствуют.

## Этап 4: Укрепление data/domain слоя

- [ ] Пройти catch-блоки репозиториев и use case, где есть только `logger.e`/`Result.failure` без `crashReporter.logException`.
- [ ] Нормализовать обработку unexpected exceptions.
- [ ] Убедиться, что `handleResponseError` и парсинг ошибок не теряют контекст.
- [ ] Добавить/обновить custom keys для Crashlytics (endpoint, feature, action, http code, user auth state).
- [ ] Проверить privacy:
- [ ] Не отправлять PII (email, токены, message body, персональные данные).

## Этап 5: Тестирование контура ошибок (TDD)

- [x] Для `ChatViewModel` добавлены unit-тесты паттерна `ViewModel error handling` (вызовы `logger`, `crashReporter.logException`, событие в `UserNotifier`).
- [ ] Контрактные тесты для репозиториев:
- [ ] `IOException` и `HttpException` обрабатываются корректно.
- [ ] Unexpected exception не теряется.
- [ ] Добавить smoke-тест "искусственный non-fatal" в debug/internal сборке для быстрой валидации Firebase.

## Этап 6: LeakCanary (базовое внедрение)

- [ ] Подключить LeakCanary только для debug/internal варианта.
- [ ] Включить автоматический экспорт результата анализа утечек (log/report/artifact).
- [ ] Настроить игнор-лист известных framework false-positive утечек.
- [ ] Зафиксировать критичные сценарии для проверки:
- [ ] `parks -> map -> other tab -> back`.
- [ ] `messages -> chat -> back` циклы.
- [ ] `profile/edit screens` циклы открытия/закрытия.

## Этап 7: Автоматизация утечек без ручных действий

- [ ] Создать инструментальный regression-сценарий, который:
- [ ] Проходит 10-20 циклов по критичным экранам.
- [ ] Триггерит GC.
- [ ] Проверяет, что LeakCanary не нашел новых подтвержденных утечек.
- [ ] Добавить отдельную команду в Makefile для CI, например `make memory-check`.
- [ ] Добавить nightly job (или pre-release job), которая запускает `memory-check`.
- [ ] Сохранять отчет как артефакт с коротким summary.

## Этап 8: CI и релизный gate

- [ ] Добавить обязательные проверки перед релизом:
- [ ] `testDebugUnitTest`.
- [ ] `lintDebug` (с исправлением текущих lint errors).
- [ ] `bundleRelease`.
- [ ] Отдельно в релизном пайплайне фиксировать `uploadCrashlyticsMappingFileRelease`.
- [ ] Добавить автоматический post-build smoke:
- [ ] Проверка, что mapping загружен.
- [ ] Проверка, что test non-fatal виден в Firebase (для internal канала).

## Этап 9: Наблюдаемость после релиза

- [ ] Настроить минимальный runbook (1 страница):
- [ ] Куда смотреть в Firebase в первые 24/72 часа.
- [ ] Какие алерты считать критичными.
- [ ] Как быстро отключить проблемный флоу (если возможно конфигом/фичефлагом).
- [ ] Установить простые SLO для маленькой аудитории:
- [ ] Crash-free users >= 99.5%.
- [ ] Новые fatal issues P0 закрываются за 24 часа.
- [ ] Повторяющиеся non-fatal P1 — в ближайшем минорном релизе.

## Этап 10: Ответ на вопрос про "полностью без ручной проверки"

- [x] Зафиксирован вывод: ручной контроль можно сократить за счет CI/nightly и авто-отчетов, но 100% автоматизация недостижима; практическая цель — 90-95% автопроверок и ручной triage только по красным кейсам.
