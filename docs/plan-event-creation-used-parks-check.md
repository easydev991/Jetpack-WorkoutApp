# TDD-план: проверка площадок перед созданием мероприятия

## Цель

Реализовать в Android поведение, аналогичное iOS:

- авторизованный пользователь нажимает создание мероприятия во вкладке `Events`;
- если у него нет площадок, где он тренируется, форма создания не открывается;
- вместо этого показывается алерт;
- из алерта можно перейти в первую вкладку bottom nav: `Parks`.

Ориентир по iOS:

- `SwiftUI-WorkoutApp/SwiftUI-WorkoutApp/Screens/Events/EventsListScreen.swift`
- ключ локализации сообщения: `Alert.EventCreationRule`

Эталонный текст сообщения:

- `ru`: “Чтобы создать мероприятие, нужно указать хотя бы одну площадку, где ты тренируешься”
- `en`: “To create an event, you need to specify at least one park where you train”

## Правило реализации

План должен выполняться строго по TDD:

1. `Red`: сначала пишем падающий тест.
2. `Green`: затем минимальную реализацию для прохождения теста.
3. `Refactor`: после зелёных тестов приводим код к финальному виду.

Нельзя сначала делать UI/навигацию, а потом дописывать тесты постфактум.

## Открытые вопросы

- [ ] Достаточно ли `User.hasUsedParks` (`area_count`) как единственного правила, или продукт ожидает проверку по фактически загруженному списку площадок пользователя.

## Риски

- Риск по навигации теперь ниже: достаточно стандартного перехода в top-level вкладку `Parks`.
- Если `currentUser` обновляется с задержкой, поведение FAB может кратковременно опираться на устаревшее состояние; это обязательно нужно фиксировать тестом.
