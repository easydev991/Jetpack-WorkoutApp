- **UI Layer:** Реализован `JournalEntriesScreen` с AppBar, Pull-to-Refresh, LazyColumn, навигацией, безопасными зонами

**Архитектурные решения:**

## Доработки: Запрет удаления первой записи в дневнике ✅

Реализована проверка, что первую запись в дневнике (с минимальным id) нельзя удалить, аналогично iOS-проекту. Сервер не позволяет удалить самую первую запись, поэтому кнопка удаления скрыта для первой записи в контекстном меню.

### Выполненные этапы

- **Domain & Data Layer:** ✅
  - Добавлен метод `getMinEntryId(journalId: Long): Long?` в `JournalEntryDao`
  - Реализован метод `canDeleteEntry(entryId: Long, journalId: Long): Boolean` в `JournalEntriesRepository`
  - 3 unit-теста

- **Domain Layer:** ✅
  - Создан интерфейс `ICanDeleteJournalEntryUseCase` и `CanDeleteJournalEntryUseCase`
  - 3 unit-теста

- **Presentation Layer:** ✅
  - Добавлено в `JournalEntriesUiState` поле `firstEntryId: Long?`
  - Добавлен метод `observeEntries()` для вычисления минимального id записей
  - Добавлен метод `canDeleteEntry(entryId: Long): Boolean` в `IJournalEntriesViewModel` и `JournalEntriesViewModel`
  - Обновлен `AppContainer` с добавлением `ICanDeleteJournalEntryUseCase`
  - 2 unit-теста

- **UI Layer:** ✅
  - Обновлен `JournalEntriesScreen` - кнопка удаления скрыта для первой записи
  - В `EntriesList` добавлен параметр `canDeleteEntry` для фильтрации доступных действий
  - Добавлен UI-тест для проверки отсутствия кнопки удаления для первой записи
  - Обновлен `FakeJournalEntriesViewModel` с методом `canDeleteEntry()`

**Архитектурные решения:**
- Проверка возможности удаления выполняется через callback `canDeleteEntry(entryId: Long): Boolean` в UI
- UI сам фильтрует доступные действия для каждой записи, что обеспечивает гибкость
- Минимальный id вычисляется в ViewModel при получении списка записей из Flow
- Для UI тестов используется простая callback-функция без реального вызова Use Case
