## Описание экрана

Экран списка записей в дневнике пользователя, который:
- Принимает идентификаторы пользователя (`userId`) и дневника (`journalId`) через path-параметры навигации
- Принимает название дневника (`journalTitle`) через query-параметр навигации (для безопасности при наличии спецсимволов в названии)
- Загружает список записей с сервера через API endpoint `SWApi.getJournalEntries(userId, journalId)`
- НЕ кеширует записи в локальной БД (аналог iOS-версии - записи только с сервера)
- Отображает список записей через компонент `JournalRowView` с режимом `JournalRowMode.ENTRY`
- При пустом списке записей показывает пустой экран без заглушки (аналог iOS-версии)
- Поддерживает Pull-to-Refresh для обновления данных (без блокировки контента)
- При первой загрузке показывает полный экран загрузки `LoadingOverlayView`
- При ошибке показывает `ErrorContentView`
- Корректно обрабатывает безопасные зоны экрана (как в `JournalsListScreen`)
- Кнопка "Назад" для возврата на экран списка дневников
- На первой итерации контекстное меню логирует действия в консоль, нажатия на записи ничего не делают

## Референсы

- iOS-версия: `SwiftUI-WorkoutApp/Screens/Profile/Journals/JournalEntriesScreen.swift`
- Экран списка дневников: `JournalsListScreen.kt` - паттерн реализации UI, безопасные зоны, Pull-to-refresh
- UI компонент: `JournalRowView.kt` - компонент для отображения записей с режимом `JournalRowMode.ENTRY`

**Важно:** Сервер возвращает `journalId` и `authorId` как `Int?`, но в доменной модели используем `Long?` для консистентности с другими ID в проекте. В маппере делаем явное приведение `.toLong()`.

- Кнопка создания новой записи НЕ реализована (аналог iOS-версии - кнопка только для владельцев)

### Архитектурные решения и важные замечания

**1. Использование query-параметра для journalTitle:**
- Используем `journal_entries/{userId}/{journalId}?title={journalTitle}` вместо path-параметра
- Причина: избежание падений при наличии спецсимволов в названии дневника (`/`, `?`, `#`)
- Query-параметр автоматически кодируется через `Uri.encode()` при навигации

**2. Приведение типов ID (Int → Long):**
- Сервер возвращает `journalId` и `authorId` как `Int?`
- В доменной модели используем `Long?` для консистентности с другими ID в проекте
- В маппере выполняем явное приведение `.toLong()` с null-safe проверкой

**3. Форматирование даты:**
- Даты хранятся в доменной модели в формате ISO (например, `2023-10-10T12:00:00`)
- Форматирование выполняется в UI при отображении через `DateFormatter.formatDate()`
- В маппере даты НЕ форматятся

**4. Отсутствие Repository слоя:**
- На первой итерации Use Case вызывает API напрямую (Direct UseCase → API)
- Причина: ускорение разработки, отсутствие кеширования (аналог iOS-версии)
- Примечание: в будущем можно будет добавить Repository без переписывания Use Case

**5. Паддинги в Scaffold:**
- Используем `contentWindowInsets = WindowInsets(0, 0, 0, 0)` и `parentPaddingValues` (для BottomNavigationBar)
- Важно: корректно суммировать `parentPaddingValues` и `innerPadding` в `ContentScreen`
- Это предотвращает "заезд" контента под системные бары или обрезание снизу

**6. Маппинг полей для UI:**
- `JournalRowView.title` → `entry.authorName` (имя автора записи)
- `JournalRowView.dateString` → отформатированная дата через `DateFormatter`
- Это отличается от логики для списка дневников, где `title` - это название дневника

**7. Кодирование и декодирование query-параметра:**
- В функции `createRoute` используем `Uri.encode(journalTitle)` для явного кодирования спецсимволов
- Это предотвращает поломку маршрута при наличии `/`, `?`, `#`, `&` в названии дневника
- В `RootScreen` декодирование НЕ требуется - Navigation Compose автоматически декодирует query-параметры через `Uri.decode()`
